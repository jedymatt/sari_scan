# Trash (replaces Archive) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace customer "Archive" with "Trash": soft-delete that is restorable and auto-purged 30 days after being trashed, with trashed customers view-only and no hard delete on active customers.

**Architecture:** Rename the existing `archivedAt` soft-flag to `deletedAt` (redefining the unmerged v2 schema in place — no v3). Add `purgeExpiredTrash()` in the db layer, triggered on app startup and when opening the Trash tab (no backend/scheduler exists). A pure `daysUntilPurge()` helper drives the "Deletes in N days" countdown. UI: the Active/Archived filter becomes Active/Trash; active customers get Move-to-Trash (with UNDO snackbar); trashed customers are view-only with Restore / Delete-permanently.

**Tech Stack:** Flutter, Drift (`^2.31.0`) + `drift_dev`/`build_runner`, `sqlite3` (tests), `intl` (`phpFormat`, date formats), Flutter gen-l10n (en + ceb).

- [ ] **Step 3: Create `lib/core/trash.dart`**

```dart
/// How long a trashed customer is retained before permanent deletion.
const trashRetention = Duration(days: 30);

/// Whole days remaining before a customer trashed at [deletedAt] is purged,
/// evaluated relative to [now]. Clamped to zero once the cutoff has passed.
int daysUntilPurge(DateTime deletedAt, DateTime now,
    {Duration retention = trashRetention}) {
  final remaining = retention - now.difference(deletedAt);
  return remaining.isNegative ? 0 : remaining.inDays;
}
```

```dart
/// Permanently deletes customers (and their entries) that have been in the
/// trash longer than [retention]. Returns the number of customers purged.
Future<int> purgeExpiredTrash({Duration retention = trashRetention}) async {
  final db = _getDatabase();
  final cutoff = DateTime.now().subtract(retention);
  return db.transaction(() async {
    final expired = await (db.select(db.customers)
          ..where((c) => c.deletedAt.isSmallerThanValue(cutoff)))
        .get();
    if (expired.isEmpty) return 0;
    final ids = expired.map((c) => c.id).toList();
    await (db.delete(db.utangEntries)..where((e) => e.customerId.isIn(ids)))
        .go();
    await (db.delete(db.customers)..where((c) => c.id.isIn(ids))).go();
    return ids.length;
  });
}
```

## Notes on the 30-day purge

`purgeExpiredTrash()` runs on app startup (`main`) and when the Trash tab opens, so on any given run, customers trashed more than 30 days ago are removed. There is no background timer — a customer trashed 30 days ago while the app is closed is purged the next time the app launches or the Trash tab is viewed, which satisfies "permanently deletes after 30 days" for this offline, single-device app.

## Known minor limitation: UNDO refresh

`_moveToTrash` pops back to the list (which reloads and correctly drops the customer from Active). The "Moved to trash" snackbar with UNDO is shown by the root `ScaffoldMessenger`, so it persists over the list. Tapping UNDO restores the customer in the database immediately, but the already-displayed Active list is not automatically reloaded — the restored customer reappears the next time the list loads (e.g. toggling the Active/Trash filter or reopening the page). The data is always correct; only the immediate visual is briefly stale. This is acceptable for this offline app; a future refinement could refresh the list after UNDO.
